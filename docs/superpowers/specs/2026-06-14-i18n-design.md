# I18n Design — EN/FR Support

**Date:** 2026-06-14  
**Status:** Approved  
**Library:** next-intl

## Overview

Add English/French internationalization to geo-compass-front-end using next-intl. Language selection via URL prefix (`/en/...`, `/fr/...`). Middleware auto-detects browser locale on first visit and redirects accordingly.

## Architecture

### Routing

URL prefix strategy. All routes get a locale prefix:

- `/en/` — English (default)
- `/fr/` — French

Middleware (`middleware.ts`) handles:
- First visit: reads `Accept-Language` header, redirects to `/en` or `/fr`
- Subsequent visits: URL prefix is authoritative
- Passes locale to next-intl's request pipeline

### File Structure Changes

```
app/
  [locale]/                     ← new locale segment wraps everything
    layout.tsx                  ← was app/layout.tsx; sets html lang attr dynamically
    (app)/
      layout.tsx
      page.tsx
      historic/page.tsx
      prompts/page.tsx
    (auth)/
      layout.tsx
      login/page.tsx
      register/page.tsx

messages/
  en.json                       ← English translations
  fr.json                       ← French translations

i18n/
  routing.ts                    ← locales array, defaultLocale
  request.ts                    ← per-request locale + messages resolution

middleware.ts                   ← next-intl middleware config

components/layout/
  locale-switcher.tsx           ← new EN/FR toggle component
```

### New Config Files

**`i18n/routing.ts`** — defines supported locales and default:
```ts
export const locales = ['en', 'fr'] as const
export type Locale = (typeof locales)[number]
export const defaultLocale: Locale = 'en'
```

**`i18n/request.ts`** — resolves locale and loads messages per request:
```ts
import { getRequestConfig } from 'next-intl/server'
import { routing } from './routing'

export default getRequestConfig(async ({ requestLocale }) => {
  const locale = (await requestLocale) ?? routing.defaultLocale
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
```

**`middleware.ts`** — next-intl routing middleware at project root.

## Translation Files

Namespaced JSON. All user-visible strings extracted from components.

### Namespaces

| Namespace | Covers |
|---|---|
| `auth.login` | Login page strings |
| `auth.register` | Register page strings |
| `auth.form` | Form field labels, placeholders, errors |
| `nav` | Sidebar and mobile nav labels |
| `dashboard` | Dashboard page and card strings |
| `prompts` | Prompts page strings |
| `common` | Shared strings (loading, errors, etc.) |

### Sample structure

```json
{
  "auth": {
    "login": {
      "title": "Welcome",
      "subtitle": "Sign in to access your AI brand rankings dashboard.",
      "noAccount": "Don't have an account?",
      "createOne": "Create one"
    },
    "register": {
      "title": "Create an account",
      "subtitle": "Sign up to access your AI brand rankings dashboard."
    },
    "signOut": "Sign out"
  },
  "nav": {
    "dashboard": "Dashboard",
    "history": "History",
    "prompts": "Prompts"
  }
}
```

## Component Usage

**Server components** (pages, layouts):
```ts
import { getTranslations } from 'next-intl/server'

const t = await getTranslations('auth.login')
return <h1>{t('title')}</h1>
```

**Client components** (nav, forms, interactive UI):
```ts
import { useTranslations } from 'next-intl'

const t = useTranslations('nav')
return <span>{t('dashboard')}</span>
```

Translation keys are fully typed — TypeScript errors on missing or misspelled keys.

## Language Switcher

`components/layout/locale-switcher.tsx` — client component.

- Renders `EN` | `FR` buttons
- Active locale visually highlighted (matches existing design system)
- Uses next-intl `useRouter` + `usePathname` to swap locale while preserving current path (e.g., `/fr/historic` → `/en/historic`)
- No page reload

Placement:
- `AppMenu` sidebar footer, beside sign out button
- `MobileBottomNav`, alongside nav items

## Navigation

Auth redirects (`router.push('/login')`) updated to use next-intl's `createNavigation` helper, which auto-prefixes the current locale. No manual locale threading in `signOut` handlers.

```ts
import { createNavigation } from 'next-intl/navigation'
import { routing } from '@/i18n/routing'

export const { Link, useRouter, usePathname, redirect } = createNavigation(routing)
```

All `next/link` and `next/navigation` imports in app code replaced with these typed, locale-aware equivalents.

## Scope

All hardcoded user-visible strings in:
- `app/(auth)/login/page.tsx`
- `app/(auth)/register/page.tsx`
- `components/layout/app-menu.tsx`
- `components/layout/mobile-bottom-nav.tsx`
- `components/dashboard/*.tsx`
- `components/prompts/*.tsx`
- Auth form components (`components/auth/`)

## Out of Scope

- Backend API responses (not translated)
- Error messages from the API
- Date/number formatting (can be added later via next-intl's format helpers)
